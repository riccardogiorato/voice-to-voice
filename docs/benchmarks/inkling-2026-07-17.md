# Inkling versus Classic voice pipeline

- Date: 2026-07-17
- Region: Vercel `iad1`
- Fixture: `test-fixtures/hello-16k.pcm`, 1.83 seconds, streamed in real time
- Runs: three successful full WebSocket turns per pipeline

Both pipelines passed the transcript, assistant text, TTS audio, RMS, and
latency-budget assertions on every measured run.

| Metric from `audio.commit` | Classic samples | Classic median | Inkling samples | Inkling median |
| --- | ---: | ---: | ---: | ---: |
| Visible user text | 489, 503, 2,154 ms | 503 ms | 1,205, 1,424, 1,752 ms | 1,424 ms |
| First TTS audio | 2,303, 2,630, 3,754 ms | 2,630 ms | 2,121, 2,466, 3,714 ms | 2,466 ms |
| TTS complete | 8,727, 9,229, 9,644 ms | 9,229 ms | 7,835, 8,075, 13,720 ms | 8,075 ms |

Inkling was mixed rather than universally faster. Its median first audio was
164 ms faster and its median completed turn was 1,154 ms faster, but its visible
transcript arrived 921 ms later. One Inkling TTS completion outlier also made
its mean total latency slower than Classic (9,877 ms versus 9,200 ms), so the
median full-turn improvement should not be treated as conclusive from this
small sample.

The `TTFT (assistant)` number is not directly comparable. Classic emits a
transcript before starting a separate reply request. Inkling returns transcript
and reply together, so the harness observes its assistant text immediately
after `transcript.final`.

Commands:

```bash
bun run test:voice -- <preview-url> --pipeline classic
bun run test:voice -- <preview-url> --pipeline inkling
```
